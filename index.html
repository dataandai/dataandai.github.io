01 · MACI — governing the agent layer
★ 44 tests incl. adversarial tenant-isolation cases
★ Terraform-first · one-command deploy & destroy
★ EU AI Act–aligned control patterns

My consulting work is concentrated in environments where client identity cannot be disclosed: regulated financial institutions and public-sector organizations in the EU. What these engagements share is not a model problem — it is a control problem. Every enterprise team can call an LLM; what blocks agentic workflows from production are the questions security, legal and audit keep asking: can we prove Tenant A never touched Tenant B's knowledge base? Can an operator shut an agent down — per tenant, per agent, per tool — before an incident becomes a compliance event? Can every model, tool and policy decision be audited tamper-evidently after the fact?

MACI (Minimal Agent Control Infrastructure) is my answer to that problem class, rebuilt in the open on AWS-native primitives so the patterns can be inspected, deployed and challenged without touching any client system or data. Where NDAs prevent me from showing the systems, MACI shows the engineering.

"The model can remain probabilistic; the operational boundary around it must be deterministic." — tenant identity, tool access, budgets, audit and failure handling are enforced by software policy, never by prompt instructions.
Design decisions
  • Identity from the trust boundary, not the model — tenant context derives exclusively from validated JWT claims and Bedrock sessionAttributes; a model asked to "act as tenant B" has no code path to do so
  • Fail-closed contracts everywhere — strict Pydantic schemas with extra="forbid" on every request and tool; unknown fields are rejected, not ignored
  • Per-operation authorization, not just tool allowlisting — an allowed tool still cannot touch another tenant's resources
  • Humans own the high-risk path — financial actions route through an approval queue; kill switches exist at global, tenant, agent and tool scope; every agent has a revocable identity with a named human custodian
  • Audit that survives an incident review — hashed, tamper-evident events with optional S3 Object Lock archive; OTel-shaped traces exported back into evaluation sets
Regulatory alignment — engineering view
Record-keeping & traceability
Hash-chained audit trail with immutable S3 Object Lock archive — the engineering counterpart of EU AI Act logging obligations
Human oversight
Approval workflow, agent custodianship and scoped kill switches map to effective-intervention requirements
Robustness & cybersecurity
Per-step guardrail checks on input, retrieved context and output; fail-closed schema validation throughout
Risk management
Documented threat model plus adversarial tests for tenant impersonation, tool misuse and RAG isolation
Data governance
Per-tenant policy engine, resource-ownership checks and knowledge-base isolation gateway — GDPR-aware separation by construction
Operational resilience
Usage ledger, budget caps and tenant-scoped circuit breakers, in line with DORA-style expectations for financial clients

Engineering interpretation, not legal advice — compliance certification remains the deployer's responsibility, a distinction I also insist on with clients.

Evidence
  • 44 passing tests covering tenant impersonation, tool tenant-override, RAG isolation, approval-bypass attempts, guardrail intervention, graph safe-stop and idempotency — runnable with zero AWS credentials, since every cloud call sits behind a mockable gateway
  • Guided first-deploy lab: from empty AWS account to smoke-tested API — plan/apply, demo tenant seeding, Cognito user + JWT, API smoke test — and a one-command destroy
  • Honest limitations, documented: threat model, production-readiness gates and environment-specific gaps are stated explicitly, because a reference that overclaims is worse than none

02 · AIBrix Tenant Gateway — governing the serving layer

High-throughput serving stacks like AIBrix/vLLM solve scheduling and GPU utilization; they do not solve tenancy. For clients running private or sovereign LLM infrastructure, the recurring question is the same one MACI answers at the agent layer, asked at the serving layer: can an adversarial client cross a tenant boundary? This gateway is the open replica of that boundary: dual tenant resolution (Host domain + JWT claim — both must agree), fail-closed model and LoRA adapter policy, spoofed-header stripping with trusted server-side re-injection, tokenizer-backed Redis sliding-window quotas, tamper-evident billing references, and a TTFT-metered streaming proxy — deployable in three modes from laptop to full GPU EKS stack, with the production gaps documented instead of hidden.

  • Spoofed routing headers (x-tenant-id, external-filter, config-profile) are stripped and re-derived from validated identity — a tenant-A token on a tenant-B domain is denied
  • Quota estimation uses a real tokenizer and ZSET sliding-window Lua scripts — no len(text)//4 heuristics in production-like modes
  • Strict Pydantic contracts reject unknown and vendor-specific fields fail-closed before anything reaches the serving pool

03 · Governed Text-to-SQL Agent — governing analytics access

Letting an LLM write SQL against a warehouse is easy; letting a non-technical manager do it safely is an engineering problem. This agent answers plain-language business questions over public BigQuery retail data under the same thesis — the model may be probabilistic, the boundary must be deterministic: sqlglot-validated single-SELECT SQL against allow-listed tables, PII blocked at three layers before it can reach a prompt or a report, server-side byte-billing caps, owner-scoped deletes behind exact-token human confirmation, bounded self-healing, an execution-based eval suite with p95 latency and error rates from live telemetry, and a production N8N automation workflow wrapping the agent for event-driven and scheduled operation.

  • 94 passing tests · 10/10 execution-based evals — runnable locally with a deterministic stub LLM and mock BigQuery, no cloud credentials required
  • PII defense in depth: generation-time instruction → SQL projection rejection (incl. SELECT *) → deterministic row masking → regex safety net on final prose
  • Destructive operations are previewed, owner-scoped and expire — replying "yes" cancels; only the exact token CONFIRM DELETE executes
